02 La redondance n°1 — deux moteurs mid-word

⚑ L'escalade F1/F2/F3 est du code mort dans la config shippée

Le bloc mid-word (PredictorViewModel.swift:851) teste if midWordLongGhostEnabled — ON par défaut — et return dans la branche long-ghost (ligne 903). La branche escalade (midWordEscalate(), ligne 905) n'est atteinte que si SOUFFLEUSE_LONGGHOST_OFF est posé.

// PredictorViewModel.swift — bloc mid-word (résumé)
if useMidWordEscalation || useAfterSpaceLongGhost {
    if midWordLongGhostEnabled {          // ON par défaut
        lg = await runtime.midWordLongGhost(...)   // ← chemin VIVANT
        ... return                            // (ligne 903)
    }
    esc = await runtime.midWordEscalate(...)   // ← F1 greedy+dico
      ... F2 : K branches stochastiques + vote modal
      ... F3 : wordCompleter.commonCompletion()  // fallback dico
}                                             // tout ce bloc dead par défaut

Concrètement, dorment par défaut : ModelRuntime.midWordEscalate() (F1/F2), la logique de branches/vote, et le fallback F3 (commonCompletion) — soit plusieurs centaines de lignes gardées uniquement comme filet A/B pour rebasculer si le long-ghost déçoit.

03 Redondances secondaires

  • L0 word-completer (système) — wordCompleterEnabledRuntime est OFF par défaut (SOUFFLEUSE_WORDCOMPLETER requis). La branche L0 « word-complete » de routeInstant ne tire jamais en prod. Le type WordCompleter ne sert plus qu'au F3 mort (et au mid-word healing). SuggestionPolicy+Tuning.swift:47 · SuggestionPolicy.swift:991
  • Paires « jumelles » — défendables : cache hit + undo-as-ghost, et corpus fast-path + history exact. Elles se ressemblent mais chacune gère une frontière distincte (mid-mot vs après-espace) ou une source distincte. À garder, mais c'est ce qui densifie visuellement l'arbre et donne l'impression de « trop de voies ».
  • INDIRECT bien séparé : n-gram / few-shot / typo-fix / KV-reuse ne sont pas des voies de génération — les compter comme telles gonfle artificiellement le total à 7+.

04 Arbre cible (si l'A/B long-ghost est conclu)

Hypothèse : le long-ghost a gagné l'A/B et « le LLM contextuel possède le mid-word » (rationale déjà écrit dans le code). Alors on retire l'échafaudage :

predict(prefix)
│ GATES (≥3 chars · canGenerate · pas mid-mot bloquant)
│
├─ INSTANT DIRECT
│   ├─ L0 lexique appris        // mid-mot
│   ├─ L1 corpus fast-path      // prior 0.92
│   └─ L1 historique exact      // après-espace
│
├─ CACHE DIRECT
│   └─ hit + undo-as-ghost
│
├─ LONG-GHOST  (mid-mot / après-espace, instant vide)   // 1 moteur, plus 2
│
└─ STREAMING LLM (L2)  + Relevance Gate

— SUPPRIMÉ : midWordEscalate() F1/F2, F3 commonCompletion
— SUPPRIMÉ : branche L0 word-completer (routeInstant)
— inchangé : n-gram · few-shot · typo-fix · KV-reuse (INDIRECT)

Résultat : 7 voies affichées → 4 blocs, suppression du double moteur mid-word et du completer mort. Le front-loading de confiance (sources sûres avant LLM) — le cœur du produit — est intact.

Décision à prendre (pas automatique)

L'escalade et le word-completer sont des kill-switches A/B volontaires. Les retirer suppose que tu considères l'A/B clos. Si tu veux garder l'option de rebascule, on les laisse — mais on peut au minimum les marquer explicitement « legacy / A/B » et sortir midWordEscalate du chemin chaud pour alléger la lecture. Avant suppression : vérifier les autres usages de WordCompleter (mid-word healing) pour ne pas casser le healing.
