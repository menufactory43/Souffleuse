<objective>
Migrer le `container.perform` body (~410 LOC) depuis PVM vers `ModelRuntime.generate(request:cache:onChunk:) async -> StreamMetrics`. **Derrière feature flag** `SOUFFLEUSE_USE_MODEL_RUNTIME` (env var, default OFF).

PVM expose deux chemins :
- **`predict_legacy(...)`** (default) — body actuel intact, copié verbatim depuis la version courante de PVM. Aucun changement.
- **`predict_new(...)`** — appelle `runtime.generate(...)` avec PredictRequest. Body court (~30-50 LOC wiring).

Le dispatcher au début de `predict(...)` lit l'env var et branche vers l'un ou l'autre. La sémantique observable (ghost, suggestionSource, ttftMillis, tokensPerSecond) est identique pour les deux paths.

**Sécurité opérationnelle** : si `predict_new` régresse subjectivement le ghost en dev build (04-07 manual validation), on peut désactiver le flag et revenir instantanément au legacy path. Pas de revert git nécessaire.

Risque : élevé (migration de 410 LOC cross-actor), mitigé par le flag + l'isolation du legacy path.
</objective>

Sous-plan 04-06 de la décomposition Option B du 04-05 atomique original. Cible : le code à risque (container.perform body migration). Le flag découple la migration physique du switchover comportemental.

**Stratégie** :
1. ModelRuntime.generate body = copie verbatim de PVM container.perform body, avec adaptations Sendable minimes (PredictRequest carries inputs, onChunk MainActor closure).
2. PVM.predict_legacy = renommage du body actuel de predict, intact.
3. PVM.predict_new = wiring slim qui appelle runtime.generate.
4. PVM.predict = dispatcher trivial qui lit `ProcessInfo.processInfo.environment["SOUFFLEUSE_USE_MODEL_RUNTIME"]` et branche.

Le flag est lu UNE FOIS au début de predict() — pas de re-eval mid-stream.

Cancel-on-keystroke continue de marcher via `planner.beginGeneration()` + `planner.isCurrent(token)` dans la onChunk closure — sémantique préservée des deux côtés.

KV cache decision tree : ModelRuntime.generate appelle `cache.decideExtendTrimInvalidate(...)` via `MainActor.run` depuis l'intérieur du container.perform closure (cache est @MainActor). Mapping verdict → log count :
- `.extend(n)` → `kv_cache_extend count: n`
- `.trim(n)` → `kv_cache_trim count: n`
- `.cold` → `kv_cache_invalidate count: 0`
- `.fingerprintChanged` → `kv_cache_invalidate count: 1`
- `.diverged` → `kv_cache_invalidate count: 2`
- `.bypass` ou `.identical` → no log

Replay BUILD-ONLY check à la fin : confirme que les deux paths compilent et produisent le même output sur scenarios synthétiques (sans charger MLX). L'empirical validation réelle vit en 04-07.
